# 4. Claude Code Tooling System--Developer Reference

> Architecture of 42 built-in tools plus MCP dynamic tools, Zod Schema validation, permission model, and security mechanisms. The schema and execution flow of every tool have been validated in large-scale production use.
>
> **Qwen Code comparison**: ToolSearch lazy loading (reduces system-prompt tokens by 50%+), 23 BashTool security checks, StreamingToolExecutor, 3-layer permission model

## Why Tooling-System Design Is the Core of an Agent

### Problem Definition

Tools are the only interface through which an Agent interacts with the outside world--all Agent capabilities (reading/writing files, executing commands, searching code) are implemented through tools. The design of the tooling system directly determines:

| Design decision | Impact |
|---------|------|
| Number of tools and Schema | System-prompt token usage (42 tools ~= 10K+ tokens) |
| Permission model | Security--whether the Agent can be prevented from executing dangerous operations |
| Execution architecture | Performance--whether tools run serially or in parallel |
| Schema validation | Reliability--whether invalid parameters generated by the model can be handled gracefully |

### Key Design Innovation: ToolSearch Lazy Loading

Among Claude Code's 42 tools, only **10 core tools** are always loaded into the system prompt. The remaining 25+ are activated on demand through **ToolSearch**:

```
System-prompt token usage:
  Load all 42 tools: ~15,000 tokens
  Core 10 + ToolSearch: ~6,000 tokens (60% savings)
```

When the model needs a less common tool, it calls `ToolSearch("notebook edit")` -> the system dynamically injects the matching tool Schema -> the tool is available in subsequent turns.

### Competitor Tooling-System Comparison

| Agent | Tool count | Lazy loading | Security validation | Parallel execution |
|-------|--------|---------|---------|---------|
| **Claude Code** | 42 | ✓ ToolSearch | 23 Bash security checks | StreamingToolExecutor |
| **Gemini CLI** | ~25 | — | commandSafety.ts | Wave-based scheduler |
| **Qwen Code** | ~30 | — | AST read-only detection | Agent tool parallelism |
| **Copilot CLI** | ~15 | — | Command blacklist | — |
>
> **Counting rule**: 39 = 10 core tools (always loaded) + 25 deferred tools (loaded on demand by ToolSearch) + 3 internal tools + 1 conditional tool (Windows PowerShell). MCP dynamic tools are excluded (their number is determined by MCP servers). TaskStop includes the KillShell alias, and Edit includes the `replace_all` bulk-edit mode; neither is a separate tool. Common/user-resident tools are about 10 (core tools); the rest are activated dynamically based on functional needs.

## 4.1 Architecture Overview

### 4.1.1 Tool Categories

> The following categories are based on **loading mode**. The same tool may also be restricted by Feature gates (for example, TaskCreate is gated by `isTodoV2Enabled`), but loading mode and gates are independent dimensions.

| Category | Tool count | Loading mode | Notes |
|------|--------|----------|------|
| **Core tools** | 10 | Always loaded (`alwaysLoad`) | Read, Write, Edit, Bash, Glob, Grep, Agent, TodoWrite, ToolSearch, StructuredOutput |
| **Deferred tools** | 25 | Loaded on demand by ToolSearch (`shouldDefer`) | WebFetch, WebSearch, NotebookEdit, Task\*, Cron\*, Worktree\*, RemoteTrigger, Brief, AskUserQuestion, SendMessage, Team\*, Skill, PlanMode\*, LSP, MCP-related, Config |
| **Internal tools** | 3 | Always loaded | REPLTool, SleepTool, TaskStop (including KillShell alias) |
| **Conditional tool** | 1 | Windows only | PowerShell |
| **MCP tools** | ∞ | Dynamically registered | `mcp__serverName__toolName` format; schemas are provided by MCP servers |

> **Classification vs `shouldDefer`**: In the source code, TodoWrite and TaskStop both have `shouldDefer: true`, but this document classifies TodoWrite as "core" (functionally always needed for task tracking) and TaskStop as "internal" (auxiliary functionality). This classification is based on each tool's functional role rather than the `shouldDefer` property. The actual lazy-loading logic is managed centrally by `isDeferredTool()` (`ToolSearchTool/prompt.ts:62`), and `shouldDefer` is one of its conditions.

### 4.1.2 Tool Lifecycle

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│ Registration │     │ Permission   │     │ Execution    │     │ Result       │
│ phase        │     │ checks       │     │ phase        │     │ handling     │
│ buildTool()  │────→│isEnabled()   │────→│validateInput │────→│call()        │
│ define schema│     │validateInput │     │checkPerms    │     │return        │
│ set defaults │     │checkPerms    │     │call()        │     │ToolResult    │
│              │     │              │     │              │     │persist large │
│              │     │              │     │              │     │outputs       │
└──────────────┘     └──────────────┘     └──────────────┘     └──────────────┘
```

**Registration**: The `buildTool()` factory fills in secure defaults. All 40+ tools are created through this factory (core + deferred + internal + conditional, excluding MCP dynamic tools).

**Permission-check order**:
1. `isEnabled()` — Feature-gate check
2. `validateInput()` — Parameter validation (short-circuits with an error code)
3. `checkPermissions()` — Permission-rule matching (allow/deny/ask)
4. `call()` — Actual execution

## 4.2 Tool Base-Class Architecture (Source: `Tool.ts`)

### 4.2.1 Core Interface

```typescript
type Tool<
  Input extends AnyObject = AnyObject,
  Output = unknown,
  P extends ToolProgressData = ToolProgressData,
> = {
  readonly name: string                    // Tool name (for example, 'Bash', 'Edit')
  aliases?: string[]                       // Backward-compatible aliases
  searchHint?: string                      // ToolSearch keywords, 3-10 words
  maxResultSizeChars: number               // Disk-spill threshold (characters)
  readonly shouldDefer?: boolean           // true = lazy-loaded
  readonly alwaysLoad?: boolean            // true = always in the initial prompt
  readonly strict?: boolean                // true = strict API parameter mode

  // Lifecycle methods
  call(args, context, canUseTool, parentMessage, onProgress?): Promise<ToolResult<Output>>
  description(input, options): Promise<string>
  prompt(options: {
    getToolPermissionContext: () => Promise<ToolPermissionContext>
    tools: Tools
    agents: AgentDefinition[]
    allowedAgentTypes?: string[]
  }): Promise<string>

  // Security classification
  isEnabled(): boolean                     // Feature gate (default true)
  isReadOnly(input): boolean               // Read-only operation (default false)
  isConcurrencySafe(input): boolean        // Safe to run in parallel (default false)
  isDestructive?(input): boolean           // Irreversible operation (default false)
  isSearchOrReadCommand?(input): boolean   // Search/read command marker
  isOpenWorld?(): boolean                  // Open-world tool marker

  // Validation and permissions
  validateInput?(input, context): Promise<ValidationResult>
  checkPermissions(input, context): Promise<PermissionResult>
  preparePermissionMatcher?(input): Promise<(pattern) => boolean>

  // Interrupt behavior
  interruptBehavior?(): 'cancel' | 'block' // Behavior when new user input arrives

  // Helper methods
  userFacingName?(input): string           // User-visible tool name
  getToolUseSummary?(input, result): string // Tool-call summary
  backfillObservableInput?(input): void    // Backfill observable input
}
```

### 4.2.2 ToolResult Return Type

```typescript
type ToolResult<T> = {
  data: T
  newMessages?: (UserMessage | AssistantMessage | AttachmentMessage | SystemMessage)[]
  contextModifier?: (context: ToolUseContext) => ToolUseContext
  mcpMeta?: { _meta?, structuredContent? }
}
```

- `newMessages`: A tool can inject additional messages into the conversation (for example, Bash injects a git-status message after detecting git operations)
- `contextModifier`: A tool can modify the execution context for subsequent tools (for example, SkillTool injects `allowedTools`)

### 4.2.3 ToolUseContext Execution Context

The context object passed to every tool call contains:

| Field | Purpose |
|------|------|
| `options.tools` | Currently available tool set |
| `options.mcpClients` | MCP client connection pool |
| `options.agentDefinitions` | Registered Agent definitions |
| `abortController` | Cancellation signal |
| `messages` | Complete conversation history |
| `getAppState() / setAppState()` | Global state storage |
| `readFileState` | File-read state cache (LRU, prevents duplicate reads + validates before writes) |
| `agentId` | Subagent ID (set only for subagents) |
| `contentReplacementState` | Tool-result budget tracking |

### 4.2.4 buildTool() Factory Defaults

| Method | Default value | Design intent |
|------|--------|----------|
| `isEnabled` | `() => true` | Enabled by default |
| `isConcurrencySafe` | `() => false` | Not parallel by default (safe side) |
| `isReadOnly` | `() => false` | Treated as a write operation by default (safe side) |
| `isDestructive` | `() => false` | Non-destructive by default |
| `checkPermissions` | `{ behavior: 'allow' }` | Allowed by default |
| `toAutoClassifierInput` | `() => ''` | Empty classifier input |

> **Implementation note**: Defaults are designed to "fail closed"--without declaring `isConcurrencySafe`, parallel execution is not allowed; without declaring `isReadOnly`, permission handling treats the tool as a write operation.

## 4.3 Core Tools in Detail

### 4.3.1 Bash Tool (Source: `tools/BashTool/`, 12,411 LOC)

Bash is the most complex core tool in Claude Code: 18 source files covering three subsystems--command execution, security validation, and permission modeling.

#### Zod Input Schema

| Field | Type | Description |
|------|------|------|
| `command` | `z.string()` | Command to execute |
| `timeout` | `z.number().optional()` | Timeout in milliseconds (maximum determined by `getMaxTimeoutMs()`) |
| `description` | `z.string().optional()` | Description of the command's purpose (active voice, concise) |
| `run_in_background` | `z.boolean().optional()` | Run in the background (omitted when `CLAUDE_CODE_DISABLE_BACKGROUND_TASKS` is set) |
| `dangerouslyDisableSandbox` | `z.boolean().optional()` | Skip the sandbox |
| `_simulatedSedEdit` | Internal field | Precomputed sed edit result (not exposed to the model) |

#### Execution Flow

```
command input
    │
    ├─ Does _simulatedSedEdit exist? ──→ applySedEdit() (direct file edit, bypasses shell)
    │
    └─ Shell execution path:
        │
        ├─ run_in_background=true? ──→ background task → backgroundTaskId
        │
        └─ foreground execution:
            ├─ start ShellCommand.exec()
            ├─ show progress after 2 seconds (PROGRESS_THRESHOLD_MS)
            ├─ Kairos mode: auto-background after 15 seconds (ASSISTANT_BLOCKING_BUDGET_MS)
            ├─ user can manually background with Ctrl+B
            └─ complete → post-processing
```

#### Post-Processing Pipeline

1. **Git-operation tracking**: Detects whether the command triggered git operations
2. **Semantic result interpretation**: `interpretCommandResult()` maps exit codes to semantic descriptions
3. **`.git/index.lock` detection**: Detects conflicts with git lock files
4. **Sandbox-violation annotation**: Marks operations denied by the sandbox
5. **Large-output persistence**: More than `maxResultSizeChars` (30,000 characters) -> write to the `tool-results/` directory; more than `MAX_PERSISTED_SIZE` (64 MB) is truncated
6. **`<claude-code-hint />` tag extraction**: Zero-token side channel (hidden instructions in tool output)
7. **Image-output detection**: Detects binary image data and automatically resizes it

#### Command Classification

| Category | Commands | UI behavior |
|------|------|---------|
| **Search commands** | `find`, `grep`, `rg`, `ag`, `ack`, `locate`, `which`, `whereis` | Collapsible display |
| **Read commands** | `cat`, `head`, `tail`, `less`, `more`, `wc`, `stat`, `file`, `strings`, `jq`, `awk` | Collapsible display |
| **List commands** | `ls`, `tree`, `du` | Collapsible display |
| **Silent commands** | `mv`, `cp`, `rm`, `mkdir`, `chmod`, `touch`, `ln`, `cd`, `export` | No output on success |
| **Semantically neutral** | `echo`, `printf`, `true`, `false`, `:` | No side effects |
| **Background forbidden** | `sleep` | Blocks `sleep N` (N>=2) as the first command |

#### Security-Validation Pipeline (Source: `bashSecurity.ts`, 2,593 lines)

The 23-layer validator pipeline is divided into early validators (which can short-circuit with allow) and main validators (all must pass):

**Early validators (can return allow and short-circuit)**:

| # | Validator | Function |
|---|--------|------|
| 1 | `validateEmpty` | Empty commands are safe |
| 2 | `validateIncompleteCommands` | Blocks commands starting with Tab, `-`, `&&`, `||`, `;`, `>>`, or `<` |
| 3 | `validateSafeCommandSubstitution` | Allows only heredocs of the form `$(cat <<'DELIM'...DELIM)` |
| 4 | `validateGitCommit` | Allows simple `git commit -m "..."` and blocks command substitution in commits |

**Main validators (all must pass)**:

| # | Validator | Defense target |
|---|--------|----------|
| 5 | `validateJqCommand` | Blocks `jq system()` and dangerous flags |
| 6 | `validateObfuscatedFlags` | Detects obfuscation techniques such as ANSI-C quoting, Locale quoting, empty quote pairs, and quote chains |
| 7 | `validateShellMetacharacters` | `;`, `|`, and `&` inside quotes |
| 8 | `validateDangerousVariables` | Variables next to redirects/pipes |
| 9 | `validateCommentQuoteDesync` | Quote characters in `#` comments desynchronize downstream trackers |
| 10 | `validateQuotedNewline` | Newlines inside quotes plus `#`-line exploits |
| 11 | `validateCarriageReturn` | CR causing shell-quote/bash parsing differences |
| 12 | `validateNewlines` | Newlines that can separate commands |
| 13 | `validateIFSInjection` | `$IFS` and `${...IFS...}` |
| 14 | `validateProcEnvironAccess` | `/proc/*/environ` paths |
| 15 | `validateDangerousPatterns` | Backticks, process substitution `<()`, command substitution `$()`, parameter substitution `${}`, and Zsh expansion |
| 16 | `validateRedirections` | Input/output redirection |
| 17 | `validateBackslashEscapedWhitespace` | `\<space>` parsing differences |
| 18 | `validateBackslashEscapedOperators` | `\;`, `\|`, and `\&` parsing differences |
| 19 | `validateUnicodeWhitespace` | Inconsistent parsing of Unicode whitespace |
| 20 | `validateMidWordHash` | Ambiguity between comment/literal interpretation of `#` |
| 21 | `validateBraceExpansion` | Bash brace expansion `{a,b}` |
| 22 | `validateZshDangerousCommands` | Zsh commands such as `zmodload`, `emulate`, and `sysopen` |
| 23 | `validateMalformedTokenInjection` | Unbalanced delimiters combined with command separators |

> **Implementation note**: This security-validation pipeline is Claude Code's most important security line of defense. The design idea behind the 23 validator layers is to **defend against parsing differences (misparsing) between shell-quote and bash**, rather than only defending against known attack patterns. Each validator targets a specific parser-differential scenario.

#### Permission Model (Source: `bashPermissions.ts`, 2,622 lines)

**Safe environment-variable allowlist** (~32 variables): Go/Rust/Node/Python/Locale/Terminal/Color variables can be safely stripped before permission matching.

**Explicit exclusions**: `PATH`, `LD_PRELOAD`, `LD_LIBRARY_PATH`, `DYLD_*`, `PYTHONPATH`, `NODE_PATH`, `NODE_OPTIONS`, `HOME`, `SHELL`, `BASH_ENV`, and others.

**Shell names forbidden as rule prefixes**: `sh`, `bash`, `zsh`, `fish`, `sudo`, `env`, `xargs`, `doas`, `pkexec`, and ~21 others (`bash:*` would automatically approve arbitrary code execution).

**Permission-suggestion logic**: Extracts a two-word prefix from the command (for example, `git commit`, `npm run`) to generate suggested allow/deny rules. For heredoc commands it uses the prefix before `<<`; for multiline commands it uses the first line.

#### Bash Tool Prompt (Source: `prompt.ts`)

The Bash tool prompt received by the model is assembled dynamically and contains these parts:

1. **Tool-preference overrides**: Instructs the model to prefer dedicated tools over shell commands (Glob > find, Grep > grep, Edit > sed)
2. **Working directory**: Supports persistence but does not preserve shell state
3. **Sandbox instructions**: Dynamically injects filesystem/network restrictions based on sandbox configuration
4. **Git commit/PR workflow**: Detailed parallel-command guidance, HEREDOC format, and safety rules (no `--no-verify`, no force-pushing main, no `git add -A`)

### 4.3.2 Edit Tool (Source: `tools/FileEditTool/`, 1,812 LOC)

#### Zod Input Schema

| Field | Type | Constraints |
|------|------|------|
| `file_path` | `z.string()` | Absolute path |
| `old_string` | `z.string()` | Text to replace |
| `new_string` | `z.string()` | Replacement text (must differ from old_string) |
| `replace_all` | `z.boolean()` | Default false; true = replace all matches |

#### String-Matching Pipeline (Three Fallback Layers)

| Layer | Method | Description |
|------|------|------|
| 1 | Exact match | `fileContent.includes(searchString)` |
| 2 | Quote normalization | Curly quotes `''""` -> straight quotes `'"`, then match; returns the original file text (preserving curly-quote style) |
| 3 | Desanitization mapping | Restores XML tags such as `<fnr>` -> `<function_results>` |

> **Note**: There is no fuzzy/Levenshtein matching. Matching is either exact or based on specific normalization replacements.

#### Quote-Style Preservation (`preserveQuoteStyle`)

When matching succeeds through curly-quote normalization, `new_string` automatically receives the same curly-quote style:
- Opening/closing heuristic: whitespace/`(`/`[`/`{`/dash/start of string before the quote = opening quote; otherwise = closing quote
- Apostrophe exception: letter + `'` + letter -> right curly quote (correct handling for "don't" and "it's")

#### Validation Pipeline (10 Error Codes)

| Error code | Condition |
|--------|------|
| 0 | Secret detection in Team Memory |
| 1 | `old_string === new_string` (no-op) |
| 2 | Path is in a permission-denied directory |
| 3 | File already exists (empty old_string = create attempt) |
| 4 | File does not exist (with similar-file/CWD suggestions) |
| 5 | `.ipynb` file (must use NotebookEditTool) |
| 6 | File has not been read before (`readFileState` check) |
| 7 | File was modified since the last read (mtime check; Windows falls back to content comparison) |
| 8 | `old_string` not found |
| 9 | Multiple matches but `replace_all` is false |

#### Write Safety: mtime Critical Section

```typescript
// Inside the critical section (synchronous):
const { content, mtimeMs } = readFileSyncWithMetadata(filePath)
if (readFileState.mtimeMs !== mtimeMs) throw staleError  // stale-write protection
writeTextContent(filePath, newContent)  // atomic write
// Critical section ends--no async operations between the mtime check and write
```

- Encoding detection: bytes `0xFF 0xFE` -> `utf16le`; otherwise `utf8`
- Line endings are always written as LF (preserving CRLF previously caused cross-platform corruption)
- Maximum file size: 1 GiB (`MAX_EDIT_FILE_SIZE`)

> **Note**: There is no standalone MultiEditTool in the source. `MultiEdit` is only a UI verb mapping in `bridge/sessionRunner.ts:74`; the actual implementation is the Edit tool's `replace_all: true` parameter (Source: `tools/FileEditTool/`).

#### Supported File Types

| Type | Extension | Handling |
|------|--------|----------|

| **Images** | `png/jpg/jpeg/gif/webp` | sharp compression pipeline: resize -> token-budget check -> aggressive compression -> 400x400 JPEG q20 fallback |

| Limit | Default | Source |
|------|--------|------|
| `maxSizeBytes` | 256 KB | GrowthBook `tengu_amber_wren` or hardcoded |

Same `(file_path, offset, limit)` and unchanged mtime -> returns a `file_unchanged` stub. Controlled by the GrowthBook killswitch `tengu_read_dedup_killswitch`.

### 4.3.4 Write Tool (Source: `tools/FileWriteTool/`, 856 LOC)

#### Zod Input Schema

| Field | Type | Description |
|------|------|------|
| `file_path` | `z.string()` | Absolute path |
| `content` | `z.string()` | Complete file contents |

#### Validation Pipeline

| Error code | Condition |
|--------|------|
| 0 | Secret in Team Memory |
| 1 | Path denied by permission settings |
| 2 | Existing file has not been read (`readFileState` check) |
| 3 | File was modified after being read (mtime comparison) |
| — | **New file** (ENOENT): allowed directly, with no read-first requirement |

#### Atomic Write Mode

```
Outside the critical section (async-safe):
  mkdir -p parent directory
  fileHistoryTrackEdit() (idempotent backup)

Inside the critical section (synchronous):
  readFileSyncWithMetadata() → mtime check → writeTextContent()
  └── no async operations between the mtime check and write
```

- Line endings: always LF (preserving CRLF previously caused cross-platform corruption)
- Encoding: preserves the encoding detected from the original file; new files default to `utf8`
- Prompt explicitly instructs the model: use Edit for modifications (send only the diff); use Write only for new files or complete rewrites

### 4.3.6 Agent Tool (Source: `tools/AgentTool/`, 6,782 LOC)

Agent is Claude Code's subagent system and supports four generation modes:

#### Zod Input Schema

| Field | Type | Description |
|------|------|------|
| `description` | `z.string()` | Task description (3-5 words) |
| `prompt` | `z.string()` | Task content |
| `subagent_type` | `z.string().optional()` | Agent type (omitted = fork or general-purpose) |
| `model` | `z.enum(['sonnet','opus','haiku']).optional()` | Model selection |
| `run_in_background` | `z.boolean().optional()` | Background execution |
| `name` | `z.string().optional()` | Teammate name |
| `team_name` | `z.string().optional()` | Team context |
| `mode` | `permissionModeSchema().optional()` | Permission mode |
| `isolation` | `z.enum([...]).optional()` | Isolation mode (worktree/remote) |
| `cwd` | `z.string().optional()` | Working-directory override |

#### Four Generation Modes (Descending Priority)

| Mode | Trigger condition | Isolation | Context |
|------|----------|--------|--------|
| **Teammate** | `team_name` + `name` set | tmux split pane + worktree | Separate process, message passing |
| **Remote** | `isolation: 'remote'` (Ant only) | CCR remote execution | Fully isolated |
| **Fork** | No `subagent_type` + `isForkSubagentEnabled()` | Shared process | **Inherits the full parent context** (maximum prompt-cache hit rate) |
| **Subagent** | Other cases | Shared process | Independent context |

#### Fork Subagent Mechanism

Fork is the most efficient subagent mode:

1. **Message construction**: Clone the parent's assistant message (all content blocks), construct a placeholder `tool_result` ("Fork started -- processing in background"), and append child instructions
2. **Recursion guard**: Scan for `<fork-boilerplate>` tags in messages or `querySource === 'agent:builtin:fork'`
3. **Fork rules**: Cannot fork again, cannot converse, use tools directly, report after committing, 500-word limit
4. **Output format**: `Scope:` / `Result:` / `Key files:` / `Files changed:` / `Issues:`

#### Tool Filtering

A subagent's tool set is filtered through multiple layers:

1. MCP tools (`mcp__*`) are always allowed
2. `ALL_AGENT_DISALLOWED_TOOLS` is disabled for all agents
3. `CUSTOM_AGENT_DISALLOWED_TOOLS` is disabled for non-built-in agents
4. `ASYNC_AGENT_ALLOWED_TOOLS` restricts the tool set of asynchronous agents
5. The `tools`/`disallowedTools` fields in the Agent definition further filter tools

#### Agent Definition System

Supports three sources (descending priority):

| Source | Description |
|------|------|
| **Built-in** | Hardcoded in code (Explore, Plan, etc.) |
| **Plugins** | Registered under `plugins/` |
| **User/project** | `.claude/agents/*.md` or settings.json |

**Markdown Agent format** (`.claude/agents/` directory):
- Frontmatter: `name` (required), `description` (required), `color`, `model`, `background`, `memory`, `isolation`, `permissionMode`, `maxTurns`, `tools`, `mcpServers`, `hooks`
- Body: system prompt

## 4.4 BashTool Security Pipeline in Detail (Source: `tools/BashTool/bashSecurity.ts`, 2,593 LOC)

> BashTool's security validation is one of the most complex subsystems in the Claude Code tooling system. This section analyzes the full execution flow of the 23-layer validator pipeline based on source code, for reference by other Agent developers designing shell-security mechanisms.

### 4.4.1 Validator Execution Order

Validators are divided into three phases: preflight gates -> early validators (can short-circuit with allow) -> main validators (ask/passthrough only).

#### Preflight Gates (Phase 0)

Executed before validators run; rejects directly:

| # | Check | Description |
|---|------|------|
| 0 | **CONTROL_CHARACTERS** (ID 17) | Rejects control characters `\x00-\x08`, `\x0B`, `\x0C`, `\x0E-\x1F`, `\x7F` |
| 0b | **shell-quote quote vulnerability** | Detects the `'''` pattern, exploiting incorrect handling of backslashes inside single quotes by the shell-quote library |

#### Early Validators (Can Return allow and Short-Circuit All Subsequent Checks)

| # | Validator | ID | Short-circuit condition |
|---|--------|-----|----------|
| 1 | `validateEmpty` | — | Blank command -> allow |
| 2 | `validateIncompleteCommands` | 1 | Commands starting with `-`, tab, `&&`/`||`/`;`/`>>`/`<` -> ask |
| 3 | `validateSafeCommandSubstitution` | 8 | Safe heredoc pattern `$(cat <<'DELIM'...DELIM)` -> allow |
| 4 | `validateGitCommit` | 12 | `git commit -m "simple message"` (no backslashes, no metacharacters) -> allow |

#### Main Validators (Run in Order; Can Return Only ask/passthrough)

| # | Validator | ID | Defense target |
|---|--------|-----|----------|
| 5 | `validateJqCommand` | 2/3 | Blocks `jq` `system()` calls and dangerous flags (`-f`, `--from-file`, `--slurpfile`, `-L`) |
| 6 | `validateObfuscatedFlags` | 4 | Catches flag obfuscation: ANSI-C quoting `$'...'`, locale quoting `$"..."`, empty-quote prefix `''-`, 3+ consecutive quotes |
| 7 | `validateShellMetacharacters` | 5 | Detects `;`, `|`, and `&` inside quoted arguments (for example, `find -name "foo;bar"`) |
| 8 | `validateDangerousVariables` | 6 | Blocks `$VAR` in redirection/pipe contexts |
| 9 | `validateCommentQuoteDesync` | 22 | Quotes inside `#` comments desynchronize the quote tracker |
| 10 | `validateQuotedNewline` | 23 | Newline inside a quoted string where the next line starts with `#` (bypasses stripCommentLines) |
| 11 | `validateCarriageReturn` | 7 | `\r` outside quotes (shell-quote/bash tokenization differences) |
| 12 | `validateNewlines` | 7 | Newlines in unquoted content (hidden second command) |
> **Note**: `validateCarriageReturn` and `validateNewlines` share enum ID 7 (`NEWLINES`). In the source, the `BASH_SECURITY_CHECK_IDS` enum has 23 values (IDs 1-23), but there are actually 25 `validate*` functions (including `validateEmpty` and `validateSafeCommandSubstitution`, which have no enum ID). In this document, "23-layer validators" refers to the number of enum IDs.
| 13 | `validateIFSInjection` | 11 | Any use of `$IFS` or `${...IFS...}` |
| 14 | `validateProcEnvironAccess` | 13 | Access to `/proc/*/environ` paths |
| 15 | `validateDangerousPatterns` | 8 | Backticks, process substitution `<()`/`>()`, `$()`, `${}`, zsh glob qualifiers |
| 16 | `validateRedirections` | 9/10 | `<` and `>` operators |
| 17 | `validateBackslashEscapedWhitespace` | 15 | `\ ` and `\<tab>` outside quotes |
| 18 | `validateBackslashEscapedOperators` | 21 | `\;`, `\|`, `\&`, `\<`, `\>` outside quotes |
| 19 | `validateUnicodeWhitespace` | 18 | 10 Unicode whitespace characters (`\u00A0`, `\uFEFF`, etc.) |
| 20 | `validateMidWordHash` | 19 | `#` after non-whitespace (shell-quote vs bash parsing difference) |
| 21 | `validateBraceExpansion` | 16 | `{a,b}` and `{1..5}` expansion (comma or `..` at depth 0) |
| 22 | `validateZshDangerousCommands` | 20 | `zmodload`, `emulate`, `sysopen/read/write`, `zpty`, `mapfile`, `zf_rm`, etc. |
| 23 | `validateMalformedTokenInjection` | 14 | Unbalanced delimiters combined with command separators |

> **Design point**: `validateNewlines` and `validateRedirections` are marked as `nonMisparsingValidators`; their `ask` results are delayed to avoid short-circuiting before more severe error detection.

### 4.4.2 Command Semantic Classification

BashTool semantically classifies commands, affecting permission prompts and result display:

| Classification | Commands | Purpose |
|------|------|------|
| **Search** | `find`, `grep`, `rg`, `ag`, `ack`, `locate`, `which`, `whereis` | UI marks as a search operation |
| **Read** | `cat`, `head`, `tail`, `less`, `more`, `wc`, `stat`, `file`, `strings`, `jq`, `awk`, `cut`, `sort`, `uniq`, `tr` | Read-only operation |
| **List** | `ls`, `tree`, `du` | Directory inspection |
| **Silent** | `mv`, `cp`, `rm`, `mkdir`, `rmdir`, `chmod`, `chown`, `touch`, `ln`, `cd`, `export`, `unset`, `wait` | File operations with no output |
| **Neutral** | `echo`, `printf`, `true`, `false`, `:` | No side effects |

### 4.4.3 Semantic Interpretation of Exit Codes

`interpretCommandResult()` semantically transforms exit codes for specific commands (Source: `tools/BashTool/commandSemantics.ts`):

| Command | Exit 0 | Exit 1 | Exit 2+ |
|------|--------|--------|---------|
| `grep` / `rg` | Matches found | **No matches found (not an error)** | Error |
| `diff` | No differences | **Differences found (not an error)** | Error |
| `test` / `[` | Condition true | **Condition false (not an error)** | Error |
| `find` | Success | Partial success (some directories inaccessible) | Error |

> **Implementation note**: Semantic transformation of exit 1 for grep/diff/test is a key UX decision--it prevents the model from mistaking "no matches found" for execution failure.

### 4.4.4 Timeout Configuration

| Constant | Default | Source |
|------|--------|------|
| `DEFAULT_TIMEOUT_MS` | 120,000 (2 minutes) | Can be overridden by `BASH_DEFAULT_TIMEOUT_MS` environment variable |
| `MAX_TIMEOUT_MS` | 600,000 (10 minutes) | Can be overridden by `BASH_MAX_TIMEOUT_MS` environment variable |
| Invariant | `MAX >= DEFAULT` | `Math.max(maxEnv, defaultTimeout)` |

### 4.4.5 Sandbox Mechanism

The sandbox is based on `@anthropic-ai/sandbox-runtime` (Linux uses `bubblewrap`/`bwrap`; macOS uses a sandbox profile):

```
Sandbox filesystem-isolation configuration:
├── Allow writes: ['.', getClaudeTempDir()] + Edit permission paths + worktree paths
├── Deny writes: .claude/settings.json, .claude/settings.local.json, bare git repo files
├── Deny reads: Read permission-denied paths
└── Network isolation: extract domain allowlist from WebFetch permission rules and sandbox.network.* settings
```

### 4.4.6 Environment-Variable Safety

**Safe variables** (~32 variables, automatically stripped from command prefixes before permission matching): Go, Rust, Node, Python, pytest-related variables, `ANTHROPIC_API_KEY`, locale, terminal, color configuration, and others.

**Never safe** (explicitly forbidden): `PATH`, `LD_PRELOAD`, `LD_LIBRARY_PATH`, `DYLD_*`, `PYTHONPATH`, `NODE_PATH`, `CLASSPATH`, `RUBYLIB`, `GOFLAGS`, `RUSTFLAGS`, `NODE_OPTIONS`, `HOME`, `TMPDIR`, `SHELL`, `BASH_ENV`.

### 4.4.7 Background Process Management

Background execution is not implemented by detecting `&`; it uses an explicit parameter:

1. **User specified**: `run_in_background: true`
2. **Auto-background on timeout**: if the command is in `COMMON_BACKGROUND_COMMANDS` (`npm`, `node`, `python`, `cargo`, `make`, `docker`, `webpack`, `vite`, `jest`, `pytest`, etc.), it is automatically converted to a background task on timeout
3. **Foreground blocking budget**: in assistant mode, foreground execution longer than 15 seconds (`ASSISTANT_BLOCKING_BUDGET_MS`) is automatically backgrounded

Output limit: 5 GB on disk (`MAX_TASK_OUTPUT_BYTES`); sends SIGKILL when exceeded.

## 4.5 Query Main-Loop Architecture (Source: `query.ts` + `QueryEngine.ts` + `toolExecution.ts`)

> This section describes the core loop between LLM responses and tool execution. Other Agent developers can use this architecture as a reference for designing their own agent loop.

### 4.5.1 Three-Layer Architecture

| Layer | Source | Responsibility |
|----|------|------|
| **QueryEngine** | `QueryEngine.ts` (1,296 LOC) | Session-level orchestration: owns `mutableMessages`, usage accumulation, and the `submitMessage()` async generator |
| **query()** | `query.ts` (1,730 LOC) | Core agent loop: `while(true)` `AsyncGenerator`, emitting `StreamEvent` each round |
| **Tool execution layer** | `toolExecution.ts` (1,746 LOC) + `StreamingToolExecutor.ts` (366 LOC) | Single-tool execution (permission check -> call -> result) + streaming parallel execution |

### 4.5.2 Single-Iteration Flow

```
┌─────────────────────────────────────────────────────────────┐
│  Each while(true) iteration                                 │
│                                                             │
│  1. Preprocess: tool-result budget → snip compact →         │
│     microcompact → context collapse → autocompact           │
│  2. Token-limit check (over limit with no recovery path →    │
│     terminate)                                              │
│  3. deps.callModel() [streaming Anthropic API call]          │
│     │                                                       │
│     ├─ Each streaming chunk:                                │
│     │   ├─ assistant message with tool_use → add to executor│
│     │   └─ StreamingToolExecutor immediately starts         │
│     │      concurrency-safe tools                           │
│     │                                                       │
│  4. [stream ended]                                          │
│     │                                                       │
│     ├─ no tool_use:                                         │
│     │   ├─ Stop hooks                                       │
│     │   ├─ max_output_tokens recovery (up to 3 times)       │
│     │   └─ return { reason: 'completed' }                   │
│     │                                                       │
│     └─ has tool_use:                                        │
│         ├─ execute remaining tools                          │
│         │   (getRemainingResults / runTools)                │
│         ├─ collect attachments (memory, queued commands,    │
│         │   skill discovery)                                │
│         ├─ check maxTurns / budget / abort                  │
│         └─ state = { messages: [...query, ...assistant,     │
│                      ...toolResults] } → continue           │
└─────────────────────────────────────────────────────────────┘
```

### 4.5.3 Streaming Tool Execution

When the `tengu_streaming_tool_execution2` feature gate is enabled, tools start executing while the LLM response is still streaming:

- **Concurrency-safe tools** (`isConcurrencySafe=true`) can start in parallel
- **Unsafe tools** execute serially (must wait for the previous one to complete)
- Bash errors cancel all sibling tools through `siblingAbortController`
- After the stream ends, `getRemainingResults()` drains unfinished tools

### 4.5.4 ToolResult Type

```typescript
type ToolResult<T> = {
  data: T                                           // Strongly typed tool output
  newMessages?: (UserMessage | AssistantMessage     // Inject additional messages into the conversation
    | AttachmentMessage | SystemMessage)[]
  contextModifier?: (ctx) => ToolUseContext          // Modify execution context for subsequent tools
  mcpMeta?: { _meta?, structuredContent? }           // MCP protocol metadata
}
```

### 4.5.5 Attachment Mechanism

Attachment is Claude Code's mechanism for injecting context **outside tool results** into the conversation (Source: `utils/attachments.ts`, 3,998 LOC). It supports about 60 types:

| Category | Attachment type |
|------|----------------|
| **Files** | `file`, `edited_text_file`, `edited_image_file` |
| **Memory** | `nested_memory`, `relevant_memories` |
| **Skills** | `skill_discovery`, `skill_listing` |
| **Tasks** | `todo_reminder`, `task_reminder` |
| **Modes** | `plan_mode`, `auto_mode` |
| **Deltas** | `deferred_tools_delta`, `mcp_instructions_delta` |
| **Budget** | `token_usage`, `budget_usd`, `output_token_usage` |
| **Other** | `queued_command`, `diagnostics`, `date_change`, `hook_*` |

Attachments are wrapped by `createAttachmentMessage()` as `<system-reminder>` XML tags and injected into the conversation as user messages.

### 4.5.6 Termination Conditions

| Condition | Signal |
|------|------|
| No tool_use + stop hooks pass | `{ reason: 'completed' }` |
| maxTurns exceeded | `{ reason: 'max_turns' }` |
| Budget USD exceeded | `error_max_budget_usd` |
| User interrupt (Ctrl+C) | `{ reason: 'aborted_streaming' }` |
| prompt_too_long (413) | Context compression -> retry; terminate if unrecoverable |
| max_output_tokens | Inject a "continue thinking" message, retry up to 3 times |

### 4.5.7 Context-Management Strategies (Executed by Priority)

| Strategy | Description |
|------|------|
| **Tool-result budget** | Persist oversized tool results to disk and replace them with `<persisted-output>` references |
| **Snip compact** | Remove old messages above threshold and generate a summary |
| **Microcompact** | Rewrite tool-result content in place (cache edit) |
| **Context collapse** | Collapsed view of REPL history |
| **Autocompact** | Full summarization when token count exceeds the context window |
| **Reactive compact** | Triggered by 413 errors as a fallback for proactive compression |

## 4.6 Complete Permission-System Flow (Source: `utils/permissions/`)

> The permission system is the security gatekeeper for tool execution. This section details the complete path from the `hasPermissionsToUseTool()` entry point to the final decision.

### 4.6.1 Permission-Check Flow (Phase 1: Core Decision)

```
hasPermissionsToUseToolInner()
    │
    ├─ Step 1a: deny-rule matching (entire tool level)
    │   └─ scan alwaysDenyRules from all PERMISSION_RULE_SOURCES
    │
    ├─ Step 1b: ask-rule matching (entire tool level)
    │   └─ scan alwaysAskRules
    │   └─ exception: BashTool + sandbox enabled + autoAllowBashIfSandboxed → skip ask
    │
    ├─ Step 1c: tool.checkPermissions(parsedInput, context)
    │   └─ custom per tool (BashTool: parse command → prefix rules → subcommand rules → operator checks)
    │
    ├─ Step 1d: tool returns deny → reject immediately
    ├─ Step 1e: tool requires user interaction → force prompt
    ├─ Step 1f: content-level ask rule → force prompt
    ├─ Step 1g: safety checks (bypass-immune) → force prompt
    │
    ├─ Step 2a: bypassPermissions mode → allow
    ├─ Step 2b: alwaysAllow rule match → allow
    │
    └─ Step 3: convert passthrough to ask
```

> **bypass-immune steps** (1d/1e/1f/1g): These must execute even in bypassPermissions mode. They protect `.git/`, `.claude/`, `.vscode/`, shell configuration files, Windows path bypasses, and cross-machine bridge messages.

### 4.6.2 Complete PermissionMode Enumeration

| Mode | Icon | Externally exposed | Behavior |
|------|------|----------|------|
| `default` | `''` | ✓ | Tools without a matching allow rule require user confirmation |
| `acceptEdits` | `⏵⏵` | ✓ | File edits inside the working directory are automatically allowed; others require confirmation |
| `plan` | `PAUSE_ICON` | ✓ | Read-only + plan-file writes |
| `bypassPermissions` | `⏵⏵` | ✓ | Allows all tools (except bypass-immune steps); requires `--dangerously-skip-permissions` |
| `dontAsk` | `⏵⏵` | ✓ | Converts all ask decisions to deny |
| `auto` | `⏵⏵` | ✗ (Ant-only) | AI classifier decides allow/deny; dangerous permission rules are stripped |

Shift+Tab cycle order: external users `default → acceptEdits → plan → bypassPermissions`; Ant users `default → bypassPermissions → auto`.

### 4.6.3 Auto-Mode Classifier Pipeline

When the result is `ask` and the mode is `auto`, three fast-path layers run:

1. **acceptEdits fast path**: Re-run `tool.checkPermissions()` in `acceptEdits` mode; if automatically allowed, skip the classifier (excluding Agent and REPL tools)
2. **Safe-tool allowlist**: `SAFE_YOLO_ALLOWLISTED_TOOLS` contains about 25 tools (Read, Grep, Glob, LSP, ToolSearch, TodoWrite, Task*, PlanMode, etc.)
3. **AI classifier**: `classifyYoloAction()` uses two-stage XML classification:
   - Stage 1 (fast): append "Err on the side of blocking" and parse `<block>yes/no</block>`
   - Stage 2 (thinking): deeper analysis with `<thinking>` tags
   - When the classifier is unavailable: behavior depends on `tengu_iron_gate_closed` (fail-closed = deny; fail-open = prompt)
4. **Denial limit**: >=3 consecutive denials or >=20 total denials -> fall back to user prompt

### 4.6.4 Circuit Breaker

Circuit breakers for auto mode:

- GrowthBook gate `tengu_auto_mode_config.enabled`: `enabled`/`disabled`/`opt-in`
- After trigger: disable classifier -> restore dangerous permissions -> switch to `default` mode
- Independent circuit breaker for `bypassPermissions`: exits directly when `tengu_disable_bypass_permissions_mode` triggers

### 4.6.5 Permission-Rule Format and Sources

**Rule format**:
```
ToolName                      # Tool-level rule (for example, "Bash")
ToolName(content)             # Content rule (for example, "Bash(npm install)")
ToolName(prefix:*)            # Prefix rule (for example, "Bash(git commit:*)")
```

**Rule sources** (descending priority):

| Source | Description |
|------|------|
| CLI parameters | `--allowed-tools "Bash(git commit:*),Read,Edit"` |
| User settings | `permissions.allow/deny/ask` in `~/.claude/settings.json` |
| Project settings | `.claude/settings.json` |
| Local settings | `.claude/settings.local.json` (gitignored) |
| Enterprise policy | managed/enterprise policy |
| Session rules | Rules generated during user interaction |

### 4.6.6 User Decisions Generate Persistent Rules

When the user clicks "Yes, and don't ask again for [X]":

1. Create a `PermissionUpdate` (for example, an allow rule for `Bash(npm install:*)`)
2. `persistPermissionUpdates()` writes it to the `permissions.allow` array in the corresponding settings file
3. The in-memory `ToolPermissionContext` is updated in real time
4. Rule deduplication: by normalized string comparison

## 4.8 MCP Tool Integration (Source: `tools/MCPTool/`)

### 4.8.1 Dynamic Registration

MCP tools are dynamically registered in the `mcp__serverName__toolName` format (note the **double underscores**). When each MCP server starts, its tool schema is parsed into a `Tool` instance.

### 4.8.2 MCPTool Base Class

Key properties: `isMcp: true`; `checkPermissions` is always `behavior: 'passthrough'`.

### 4.8.3 MCP Authentication Tool

**McpAuthTool**: Handles the OAuth authentication flow for MCP servers.

### 4.8.4 MCP Resource Tools

## 4.9 Cross-Tool Security Architecture

### 4.9.1 Permission System

> For the full description of the permission system, see **Section 4.6** (6 PermissionMode values, 10-step check pipeline, 3-layer Auto classifier, Circuit Breaker, and 6-level rule-source priority). The following is a cross-tool security summary.

Core principle: **bypass-immune** (steps 1a-1g, including `.git/`/`.claude/` path protection) is enforced in all permission modes; `checkPermissions` is implemented by each tool (for example, Bash's 23-layer validators and Edit's mtime critical section). The CLI parameter is `--allowed-tools` (kebab-case).

### 4.9.2 Read-Before-Write Protection

Edit and Write share `readFileState` (LRU cache, Map<path, { content, timestamp, offset, limit }> ):

1. **Must read before writing**: The file must have been read through the Read tool in the current session
2. **mtime validation**: During writes, check the file mtime against the value recorded at read time
3. **Synchronous critical section**: No async operations are allowed between the mtime check and write (prevents interleaved concurrent edits)
4. **Windows fallback**: Use content comparison when mtime is unreliable (prevents false positives from cloud sync/antivirus software)

### 4.9.3 Large-Output Persistence

Shared large-output handling pattern for all tools:

1. Tool result exceeds `maxResultSizeChars` -> write to the `tool-results/` directory
2. Return a `<persisted-output>` tag + preview snippet
3. Truncate beyond `MAX_PERSISTED_SIZE` (64 MB)
4. Path format: `tool-results/<tool-name>-<timestamp>.txt`

> **Exception**: The Read tool sets `maxResultSizeChars: Infinity` (Source: `FileReadTool.ts:342`) because its output is already size-limited by line count/file type; adding another limit would cause a read loop (Read -> persist -> need to Read again).

### 4.9.4 File-History Tracking

Edit and Write call `fileHistoryTrackEdit()` for idempotent backups, supporting undo operations.

### 4.9.5 UNC Path Security

Windows SMB paths (`\\server\share`) are intercepted in Edit, Write, NotebookEdit, and LSPTool to prevent NTLM credential leakage.

### 4.9.6 Team Memory Secret Scanning

Edit, Write, and TodoWrite detect Team Memory secrets in content (error code 0), preventing accidental leakage.

## 4.11 Design Philosophy and Architectural Trade-Offs

> This section analyzes the reasoning behind Claude Code tooling-system design decisions. Each decision point compares two common approaches and explains Claude Code's choice and engineering rationale. Information sources are source-code comments and inferences from code structure (⚠️ inferred parts are marked).

### 4.11.1 Fail-Closed Security Defaults: Why the `buildTool()` Factory Is Necessary

**Design choice**: The `buildTool()` factory function centrally manages secure defaults instead of requiring each tool to implement them itself.

**Source basis**: Comments in `TOOL_DEFAULTS` in `Tool.ts:753-767` explicitly state:

> Defaults (fail-closed where it matters):
> - `isConcurrencySafe` -> `false` (**assume not safe**)
> - `isReadOnly` -> `false` (**assume writes**)

**Comparative analysis**:

| Approach | Advantages | Disadvantages |
|------|------|------|
| **A: Central factory + fail-closed** (Claude Code choice) | New tools cannot omit security methods; `isConcurrencySafe`/`isReadOnly` default to the most conservative behavior | Must keep `TOOL_DEFAULTS` synchronized with the `Tool` type |
| **B: Each tool declares its own methods** | High flexibility | New tool authors may forget `isReadOnly()`, causing a tool to be incorrectly marked "safe" and run concurrently |
| **C: Manual expansion with `Object.assign`** | No additional abstraction layer | No type-level guarantee; different tools may use different defaults objects, causing inconsistency |

⚠️ Inference: The key reason Claude Code chose approach A is the **large number of tools** (39 explicit + MCP dynamic), which makes manual management unreliable. The `buildTool` comment (`Tool.ts:778`) explicitly says:

> The type semantics are proven by the 0-error typecheck across all 60+ tools.

### 4.11.2 Lazy Loading + ToolSearch: Token Economics

**Design choice**: ~10 core tools are always loaded; the remaining 25 tools are lazy-loaded (counted by directory; about 28+ after expanding subtools, see the formula in 4.1.1) and discovered on demand by ToolSearch.

**Source basis**: `utils/toolSearch.ts` defines three modes (`tst` / `tst-auto` / `standard`); the `tst-auto` threshold is **10%** of the context window (`DEFAULT_AUTO_TOOL_SEARCH_PERCENTAGE = 10`).

**Comparative analysis**:

| Approach | Token cost | Latency | Complexity |
|------|-----------|------|--------|
| **A: Lazy loading + search engine** (Claude Code choice) | ~10 tool schemas always in prompt; MCP tools list names only | First use requires a ToolSearch call (~1 API round) | High (requires ToolSearchTool + scoring algorithm + 6-level weights) |
| **B: Load everything** | All tool schemas consume prompt tokens (MCP may add dozens of tools) | Zero latency | Low |
| **C: Preload by scenario** | Medium | Low | Medium (must maintain scenario -> tool mapping) |

⚠️ Inference: The core motivation for approach A is **the unpredictability of the MCP ecosystem**. A project may configure three MCP servers (providing 50+ tools), and loading everything would consume a large part of the context window. The 10% threshold in `tst-auto` mode is an empirical balance point.

The ToolSearch scoring algorithm (Source: `ToolSearchTool.ts:155-260`) uses 6 weight levels (+12/+10/+6/+5/+4/+3/+2). MCP tool-name matches score higher than native tool-name matches (+12 vs +10), reflecting the stronger signal in MCP tool names that include a server prefix (for example, `mcp__github`).

### 4.11.3 Fork vs Subagent: Prompt-Cache-Driven Architecture

**Design choice**: The Agent tool supports two subagent modes--Fork (shared context) and Subagent (independent context).

**Source basis**: The JSDoc in `utils/forkedAgent.ts` explicitly states the design motivation:

> Parameters that must be identical between the fork and parent API requests to share the parent's prompt cache. The Anthropic API cache key is composed of: system prompt, tools, model, messages (prefix), and thinking config.

**Comparative analysis**:

| Dimension | Fork (shared context) | Subagent (independent context) |
|------|-------------------|----------------------|
| **Prompt cache** | ✅ Shares parent cache (API request prefix is fully identical) | ❌ Independent request, no cache sharing |
| **Context** | Clones the parent's full conversation history | Starts from a blank conversation |
| **Tool set** | Uses the parent's exact tool pool (`useExactTools: true`) | Reassembled (may have different permission mode) |
| **System prompt** | Passes already rendered bytes directly (not regenerated) | Calls the agent's `getSystemPrompt()` |
| **Permissions** | `'bubble'` mode (prompts bubble back to parent terminal) | Agent-defined or isolated |
| **Token cost** | Low (shared prefix ~= free replay) | High (full system prompt + CLAUDE.md + gitStatus) |
| **Isolation** | Low (subagent can see parent conversation) | High (fully isolated) |

⚠️ Inference: Every design detail of Fork mode--threaded rendered bytes (avoiding GrowthBook cold -> warm state changes), `useExactTools` (avoiding tool-serialization differences), inheriting thinking config (avoiding cache-key changes)--exists to achieve a **byte-identical API request prefix**. This is not merely an optimization; it treats the Anthropic prompt-cache mechanism as an architectural constraint.

The source comment (`forkSubagent.ts`) says:

> Reconstructing by re-calling getSystemPrompt() can diverge (GrowthBook cold→warm) and bust the prompt cache; threading the rendered bytes is byte-exact.

Recursion protection uses a **two-layer guard**: a `querySource` marker (survives autocompact) + scanning for the `<fork-boilerplate>` tag (survives message history), ensuring the protection is not lost even if autocompact rewrites messages.

### 4.11.4 Three-Stage Permission Pipeline: Why Validation and Permissions Must Be Separate

**Design choice**: `hasPermissionsToUseTool` is divided into 10 steps, with the core split into three phases: `validate` (steps 1a-1g, not bypassable) -> `check` (steps 2a-2b, bypassable) -> `call`.

**Source basis**: The comments for steps 1a-1g (`permissions.ts`) state:

> Safety checks (.git/, .claude/, .vscode/, shell configs) are bypass-immune — they must prompt even in bypassPermissions mode.

**Comparative analysis**:

| Approach | Security | Flexibility | Complexity |
|------|--------|--------|--------|
| **A: Unified pipeline** | Low (bypass mode may skip critical security checks) | High | Low |
| **B: Separate pipelines (Claude Code choice)** | High (bypass-immune checks always execute) | Medium | High (must maintain two independent hook paths) |

The key reason Claude Code chose approach B is the existence of `bypassPermissions` mode. If all checks ran in a single pipeline, `--dangerously-skip-permissions` would skip all safety checks, including protection for the `.git/` directory. Separate pipelines ensure that **even bypass mode cannot bypass core safety rules**.

The `checkRuleBasedPermissions` function (used by PreToolUse hooks) copies only steps 1a-1g and does not go through steps 2a-2b, further showing that this separation is intentional.

### 4.11.5 Auto Mode Three-Layer Classifier: Balancing Latency and Safety

**Design choice**: Auto mode (`--dangerously-skip-permissions`) uses a 3-layer cascading decision: acceptEdits fast path -> safe-tool allowlist -> AI classifier.

**Source basis**:

- **Layer 1** comment (`permissions.ts`): > This avoids expensive classifier API calls for safe operations like file edits in the working directory.

- **Layer 2** comment (`classifierDecision.ts`): > Does NOT include write/edit tools — those are handled by the acceptEdits fast path.

- **Layer 3** protection: sends only the `tool_use` block to the classifier, not assistant text -> assistant text is model-authored and could be crafted to influence the classifier's decision.

**Comparative analysis**:

| Approach | Latency/use | Safety coverage | Token cost/use |
|------|---------|----------|--------------|
| **A: Full AI classification** | ~500ms | High | ~200 tokens |
| **B: Static allowlist** | <1ms | Medium (cannot judge semantic safety) | 0 |
| **C: Three-layer cascade** (Claude Code choice) | ~50ms (average) | High (AI classifier fallback) | ~50 tokens (average) |

⚠️ Inference: The three-layer cascade is a **latency optimization**--most file-edit operations are caught by Layer 1 (acceptEdits check is synchronous), read-only operations are caught by Layer 2 (allowlist match), and only truly uncertain operations reach the Layer 3 AI classifier.

The **Circuit Breaker** (3 consecutive denials or 20 total denials -> fall back to interactive mode) is an additional safety net. For headless agents, exceeding the limit throws `AbortError` directly, preventing auto mode from entering a "deny -> retry" loop.

### 4.11.6 Bash 23-Layer Validators: Defending Against Shell Parser Differentials

**Design choice**: The Bash security pipeline uses 23 validator layers rather than a whitelist or sandbox-only defense.

**Source basis**: Each validator defends against a specific attack pattern. The core design principle comes from comments in `bashSecurity.ts`:

> This is an EARLY-ALLOW path: returning `true` causes bashCommandIsSafe to return `passthrough`, bypassing ALL subsequent validators. Given this authority, the check must be PROVABLY safe, not "probably safe".

The comment for validator #13 (carriage-return injection) explains the root cause:

> Parser differential: shell-quote's BAREWORD regex uses `[^\s...]` — JS `\s` INCLUDES `\r`, so shell-quote treats CR as a token boundary. bash's default IFS = `$' \t\n'` — CR is NOT in IFS. bash sees `TZ=UTC\recho` as ONE word.

**Comparative analysis**:

| Approach | Defense scope | False-positive rate | Maintenance cost |
|------|----------|--------|----------|
| **A: Command allowlist** | Narrow (can allow only known safe commands) | High (every new command requires review) | Low |
| **B: Sandbox-only defense** | Broad (kernel-level isolation) | Low | Medium (sandbox-escape risk) |
| **C: Multi-layer validation + sandbox** (Claude Code choice) | Broadest (covers parser differentials + sandbox fallback) | Medium | High (23 validator layers + 4 sandbox backends) |

⚠️ Inference: The fundamental reason Claude Code chose approach C is the **semantic difference between the JavaScript `shell-quote` library and bash/zsh parsers**. This is not a problem of defending against known attack patterns; it is a problem of two parsers producing different tokenization for the same string. A whitelist cannot defend against commands that "look safe to JS but mean something different to bash".

The comment for validator #4 (obfuscated flags), over ~300 lines of code, records the most complex attack surface:

> In bash, `"""-f"` = empty string + string `"-f"` = `-f`. This bypass works for ANY dangerous-flag check with a matching prefix permission.

Defense principle (`bashSecurity.ts`): > Defense in depth: Block PowerShell comment syntax even though we don't execute in PowerShell. Added as protection against future changes that might introduce PowerShell execution.

### 4.11.7 mtime Critical Section: Why TOCTOU Protection Must Be Synchronous

**Design choice**: FileEditTool's "read -> check -> write" sequence must complete inside a synchronous critical section, with no async yield allowed.

**Source basis**: The comment in `FileEditTool.ts` says:

> Please avoid async operations between here and writing to disk to preserve atomicity

**Comparative analysis**:

| Approach | Concurrency safety | Performance | Implementation complexity |
|------|----------|------|-----------|
| **A: File locks** | Safe | Blocks other processes | High (requires cross-platform lock mechanism) |
| **B: Optimistic concurrency + async mtime check** | Unsafe (TOCTOU window) | High | Low |
| **C: Synchronous critical section + mtime** (Claude Code choice) | Safe (no yield window) | Medium (blocks the Node event loop for ~ms) | Medium |

The root cause of TOCTOU (Time-of-Check-Time-of-Use) vulnerabilities: if there is an async yield between `readFile` and `writeFile`, another tool call (or concurrent agent) may modify the file during the yield, causing the write to be based on stale content.

Claude Code's rationale for approach C (⚠️ inference): File locks (approach A) are complex to implement across platforms (Windows/macOS/Linux) and across processes (agent subprocesses). Although a synchronous critical section blocks the event loop, file editing itself is CPU-bound string matching + file writing and typically takes only milliseconds, so the user-experience impact is negligible.

Windows-specific handling: because cloud sync and antivirus software may change file mtime without changing content, full-file reads use **content comparison** as a secondary check to avoid false positives.
